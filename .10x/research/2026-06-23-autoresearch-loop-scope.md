# 10x Autoresearch Loop Scope

## Question

What should an autoresearch loop for 10x optimize first, using numeric scores that let agents run repeated experiments against the 10x protocol without prematurely implementing machinery?

## Findings

The peer repository is primarily a methodology repo, not a reusable framework. Its durable mechanics are:

- Experiment logs come first. Each experiment has a hypothesis, method tier, prediction, status, and verdict.
- Negative, refuted, neutral, and "already optimal" results are treated as useful outcomes.
- Method tiers escalate from cheap to expensive:
  - MINE: answer from existing run artifacts.
  - MICRO: run small prompt batteries with controls, repeated samples, and cached outputs.
  - FULL: run real end-to-end evaluation only when cheaper tiers cannot answer.
- Every micro battery includes a no-guidance control so the loop can distinguish improvement, backfire, and inconclusive-by-zero.
- Automated scoring is treated as a first pass only. Manual inspection is mandatory because several previous scoring bugs were caused by template echoes, missing fixture content, and weak regexes.
- Single-run full-eval deltas under roughly 20 percent are treated as noise.
- Fixture realism is part of validity. Hand-written fixtures can distort the measured behavior; fixtures should be generated by the system under test when possible.
- A baseline that does not fail is not evidence that guidance works.
- Reports are narrative artifacts over experiment logs, not the source of truth.

For 10x, the optimization target should not start as generic task success. The 10x protocol's differentiator is disciplined behavior: ambiguity handling, durable record graph maintenance, ticket boundaries, evidence-backed claims, adversarial review, retrospective learning, and refusal to build unnecessary code.

## Candidate Scoreboard

Use a vector of continuous scores first. A single scalar can rank variants later, but only after the diagnostic scores are trustworthy.

| Score | Range | What it measures | Likely first scorer |
| --- | ---: | --- | --- |
| Outer Loop Discipline | 0-100 | Detects ambiguous work, inspects before asking, avoids premature implementation, asks decisive questions, and offers a concrete recommendation. | Micro scenarios with tool-call and transcript scoring. |
| Record Graph Fitness | 0-100 | Creates the right `.10x/` record type, required headers, durable context, references, status coherence, and no record spam. | Filesystem scorer plus manual review. |
| Ticket Readiness | 0-100 | Tickets are scoped, bounded, acceptance-ready, non-goals explicit, and cold-start executable without guessing. | Rubric judge plus schema checks. |
| Evidence Integrity | 0-100 | Material claims are backed by recorded observations, limits are stated, and passing commands are not overclaimed. | Claim-to-evidence scorer plus reviewer sampling. |
| Scope Minimalism | 0-100 | The agent avoids speculative features, extra dependencies, broad refactors, and unnecessary files while preserving safety rails. | Diff scorer in FULL runs; transcript scorer in MICRO. |
| Closure Coherence | 0-100 | Acceptance criteria, evidence, reviews, statuses, related specs, and retrospective extraction agree before closure. | Record graph scorer plus adversarial review. |
| Human Shaping Quality | 0-100 | Questions are few, material, concrete, and paired with recommendations instead of forms or vague discovery. | Transcript judge with examples. |
| Cost Efficiency Index | Baseline 100, unbounded | Quality-adjusted cost, wall time, turns, context size, and API spend compared with a baseline. | Mining and FULL run economics. |

The first top-line index should be quality-gated, not purely additive:

```text
Core10xQuality = geometric_mean(
  OuterLoopDiscipline,
  RecordGraphFitness,
  TicketReadiness,
  EvidenceIntegrity,
  ScopeMinimalism,
  ClosureCoherence
)

10xResearchIndex = 100 * (Core10xQuality / baseline_Core10xQuality)
                   * (baseline_cost / variant_cost) ^ 0.25
```

Candidate rule: do not compare `10xResearchIndex` unless every core score is at or above an agreed floor. This keeps the loop from making agents cheaper by making them less 10x.

## Recommended Loop Shape

Start with the peer repo's loop structure, adapted to 10x records:

1. Pre-register each experiment in a dated research or experiment log before changing the 10x instructions.
2. Always run the cheapest useful tier first:
   - MINE prior sessions, transcripts, `.10x/` records, and diffs.
   - MICRO scenario batteries with no-guidance, current-10x, and candidate-10x arms.
   - FULL end-to-end coding gauntlets only for behavior that depends on real filesystem, tickets, evidence, review, or implementation.
3. Score every run numerically, but require human/manual inspection for every proposed conclusion until scorers have their own track record.
4. Preserve raw outputs and record conclusions in `.10x/research/` or `.10x/evidence/`.
5. Promote stable conclusions to specs, decisions, knowledge, or tickets only after the scores and manual review agree.

## Initial Scenario Battery Ideas

The first score battery should deliberately include traps that 10x is meant to handle:

- Ambiguous implementation request: agent should stay outer-loop, inspect records, and ask a focused question rather than code.
- "Just do it" request with missing acceptance criteria: agent should identify what is missing and propose a concrete execution contract.
- Non-trivial clear request: agent should open or use a ticket and not implement as the parent when subagent execution is required.
- Closure trap: tests pass but evidence/review/spec status is incomplete; agent should refuse closure or record limits.
- Minimalism trap: user asks for a broad abstraction where a smaller native change would solve the problem.
- Record spam trap: many possible records could be written, but only one durable research/spec/ticket record is warranted.
- Evidence overclaim trap: command output supports a narrow claim, but the agent is tempted to claim global correctness.

## Tradeoffs

- A single scalar is useful for an autonomous optimizer but dangerous as the only truth. Keep the diagnostic vector visible.
- Programmatic scoring is necessary for scale but will be wrong in subtle ways. Manual inspection is part of the methodology, not ceremony.
- Full end-to-end runs are needed eventually because 10x behavior depends on real file and record state. They should not be the first tool.
- Numeric scores can encourage record spam, excessive questioning, or over-constrained tickets unless the scorer penalizes unnecessary process.
- Cost efficiency should be optimized only after behavioral quality floors are stable.

## Open Questions

- Should the first loop optimize the `SKILL.md` text itself, a harness-native install wrapper, or a separate eval skill?
- Which harness should be the initial execution target: Codex, Claude Code, OpenCode, or a harness-agnostic transcript simulator?
- What model should be treated as the primary subject under test?
- What quality floors are acceptable before the cost index is allowed to matter?
- Should the first "control" be no 10x guidance, the current 10x skill, or a reduced 10x variant?

## Recommendations

- Start with scoring design, not implementation.
- Treat the first deliverable as a 10x autoresearch specification covering the score vector, experiment tiers, scenario suite, scorer trust model, and promotion rules. This draft now exists at `.10x/specs/10x-autoresearch-loop.md`.
- Do not create a runner until the score vector and first scenario battery are agreed.
- Use the peer repo's method discipline directly: pre-register, include controls, run repeated samples, manually inspect automated scores, and preserve negative results.

## Implementation Scoping Notes

As of 2026-06-23, implementation should not start as one broad runner. The smallest coherent route is:

1. Convert the specification into static, reviewable catalogs: score catalog, scenario catalog, experiment record template, and score artifact shape.
2. Build stdlib-only validators and first-pass scorers against those catalogs.
3. Run offline scoring against saved transcript/file fixtures before making API calls.
4. Add a MICRO runner once score outputs and manual inspection workflow are credible.
5. Add FULL harness integration only after MICRO evidence shows the score vector catches the intended failures.
6. Add reporting last, generated from records and score artifacts rather than acting as the source of truth.

Recommended defaults to resolve the current spec open questions:

- First harness: transcript/file-output simulator for MICRO scoring, then Codex as the first FULL harness because that is the current working environment.
- Primary model: the user's current Codex model for FULL runs; provider-specific model choice for MICRO should be explicitly recorded per experiment.
- No-guidance control: minimal harness defaults with no 10x text, plus current canonical `SKILL.md` as the baseline arm.
- Artifact storage: implementation code outside `.10x/`; research and evidence records under `.10x/`; raw run artifacts under `.10x/evidence/.storage/` when they support claims and `.10x/research/.storage/` for exploratory source material.
- Trust Level 3 approver: human operator only until a decision record delegates this authority.
- Human inspection time: track separately at first; do not fold into S009 until a cost policy is accepted.
- Score weights: keep fixed and transparent initially; do not learn weights until enough human verdicts exist to justify a later decision.

The remaining execution-critical ambiguity is budget. Before running MICRO or FULL experiments, the operator should set initial spend and wall-clock limits.
