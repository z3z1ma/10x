# Housekeeping

Housekeeping keeps memory useful by making it smaller, better linked, and less
authoritative over time.

## Regular checks

- prune hot memory when it stops being hot
- move older detail into observations
- archive or delete low-signal material when needed
- add or fix `[[links]]`
- keep entity entries compact
- replace duplicated canonical truth with short pointers
- move memory items that became ticket, wiki, evidence, spec, research, plan,
  initiative, or constitution truth into that owner layer, then leave only a
  support pointer if useful
- remove action-item reminders after they become real ticket work
- leave current support-only items alone when they still reduce orientation cost

## Good housekeeping question

What is the smallest memory surface that still helps continuity?

## Triggers

Run housekeeping when:

- hot memory is longer than a quick scan
- a ticket closes and support reminders no longer match the ticket truth
- a retrospective promotes learning into wiki, research, specs, plans,
  initiatives, constitution, evidence, or tickets
- memory contains a claim that future agents are likely to rely on
- memory points at records that were renamed, superseded, or retired

These are checkpoints, not a second cadence ledger. Ticket closure,
retrospective, rename/supersession cleanup, and local housekeeping are moments to
ask whether memory should be left alone, linked, marked stale, promoted, or
pruned. Do not track cleanup progress in memory as if it were ticket state.

## Stale Handling

- If the item is still useful but no longer current, mark it stale in place or
  move it to observations with a date.
- If the item now has a canonical owner, replace the detail with a pointer.
- If the item cannot be verified and no longer helps retrieval, remove it.

Marking stale is for retrieval clarity only. It does not decide whether the
underlying project truth changed; the owning canonical record decides that.

## Pruning Boundary

- Prune when the memory item is obsolete, duplicated by canonical truth, too old
  to help retrieval, or no longer worth scanning.
- Promote before pruning when the item is the only copy of a durable claim,
  accepted explanation, observation needed as evidence, intended behavior,
  investigation result, live state, or policy.
- Keep a pointer instead of detail when the owner record is easy to cite but easy
  to forget.
- Leave the item alone when it is current support recall and remains cheap to
  scan.

## Action-Item Boundary

Memory action items are reminders only.

Promote to a ticket when the item has scope, acceptance criteria, blockers,
dependencies, or a real next route. Do not track live progress in memory.
