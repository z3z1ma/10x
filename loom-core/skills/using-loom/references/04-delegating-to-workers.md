# Delegating To Workers

Use strict boundaries when execution or review leaves the operator's immediate
loop. Ticket work runs through bounded Ralph from the ticket and linked records.
The ticket is the durable contract; launch prompts are transport.

## Ticket Context Before Launch

A Loom worker run starts from graph-supported context, usually an executable
ticket and its linked records. Before subagent, headless, or manual handoff,
ensure the ticket or owning surface contains mission, context, scope, write
boundary, stop conditions, evidence expectations, and allowed record updates.

Keep the launch wrapper thin: identify the ticket or target records to read first,
name the immediate objective, restate only transient transport constraints, and
request the required output. Do not hide durable scope, acceptance, write
boundaries, or evidence expectations only in the prompt.

When worker output supports ticket state, evidence, audit, research, knowledge, or
closure, reconcile it into the owning surface.

## When To Use A Ralph Run

Use Ralph when the next worker or review pass can be bounded with a clear mission,
limited scope, known sources, read/write areas, evidence or review output, and
stop conditions.

Do not delegate unresolved operator judgment, ambiguous product intent, policy,
broad architecture, data/state modeling, or design-coherence decisions. Shape that
work first.

## Durable Run Context

A useful ticket-owned run context states:

- mission for this run
- source records, refs, evidence, diffs, or artifacts to inspect
- intended read area
- intended write area
- task boundary and non-goals
- relevant assumptions and constraints
- stop conditions
- expected evidence, review output, or validation result
- output expectations
- records or artifacts allowed to change

The run should be narrow enough that the worker can finish, stop, or report
blockage without inventing a plan. If durable context is missing, update or block
the ticket before launch instead of putting the missing truth only in the prompt.

## Worker Authority

The worker may update source or work files within scope, records or evidence
artifacts named by the ticket or launch, and the requested worker output. If new
evidence is needed but not authorized by write scope, stop or report the need
instead of widening scope.

The worker must not change high-authority direction while executing. Escalate
before changing constitution, specs, plans, or research synthesis. Knowledge
updates should be proposed, not applied, unless explicitly authorized.

Audit is a separate adversarial pass. A worker implementation report is not audit,
and a worker should not certify its own work as accepted.

## Worker Discipline

The worker should stop and report when:

- the ticket or run context no longer matches reality
- the requested change is broader than the declared scope
- required context is missing
- evidence or review output cannot be gathered
- implementation reveals a product, architecture, policy, or sequencing decision
- the safe next step belongs to another Loom surface

A worker should not silently widen scope to stay busy.

## Worker Output

The worker reports:

- what changed
- what evidence, review output, or validation result was gathered
- what was not verified or reviewed
- what remains blocked or risky
- what records were updated
- what it recommends next

## After Worker Return

The parent reads the ticket, updated records, evidence, changed files, and worker
report instead of reconstructing work from tool logs.

The parent decides whether to stop, launch another bounded run, gather evidence,
route to audit, return to shaping, update another surface, or promote knowledge.

When the result supports closure, acceptance, or durable reuse, route the claim
through bounded Ralph review before trusting it unless the consuming surface
records why audit would not add useful trust.

## Run Currency

Use a run only while its target, context, scope, and assumptions match reality. If
they change materially, update or block the ticket and launch a fresh run.
