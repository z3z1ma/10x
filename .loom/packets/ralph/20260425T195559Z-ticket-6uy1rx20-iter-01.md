# Mission

Build the smallest `open-loom` fixture that can read Loom's bundled Markdown
surfaces and test the plugin-first install shape enough to decide what must
remain fallback direct config install.

# Bound Context

Target ticket: `ticket:6uy1rx20`.

Governing plan: `plan:install-experience-harness-adapters`, Wave 3, expects an
OpenCode plugin-first investigation and explicitly says fallback direct installer
changes should wait until plugin API limits are proven.

Governing research: `research:loom-install-distribution-methods#opencode` says
OpenCode plugins can be npm packages or local file/path specs, current OpenCode
does not support Git URL plugin installs in practice, and plugin APIs may cover
rules through `experimental.chat.system.transform` but skill and command
registration are unproven.

Intended behavior: the ideal user adds one OpenCode `plugin` entry and the plugin
exposes bundled Loom rules, skills, and optional commands without a separate copy
step. Normal users should use npm package distribution. Local/manual users should
clone the repository and point OpenCode at a local file/path plugin entry. No
generated plugin output may become canonical Loom semantics.

Current implementation reality: `scripts/install-loom.sh` currently implements
OpenCode through direct copies into `~/.config/opencode/loom/rules`,
`~/.config/opencode/skills`, and `~/.config/opencode/commands`, then mutates
`opencode.json` `instructions`. `examples/adapters/opencode-rule-install/` only
documents that direct-rule install shape.

Git execution context: run in the current checkout on branch `main`, dirty with
the existing install-experience Loom record changes. Do not fetch, create remotes,
edit Git config, reset, checkout away changes, or commit.

# Task For This Iteration

Create the smallest useful `open-loom` plugin in this repository.

The preferred shape is either:

- root package metadata plus a root plugin entrypoint, if that is the smallest
  honest way to model npm package consumption of the existing top-level Loom
  Markdown corpus, or
- an `examples/adapters/open-loom-install/` fixture that documents exactly
  how the npm and local-path shapes map back to a future package.

Keep the implementation minimal. Do not add dependencies, build steps, generated
rules, generated skills, generated commands, hidden runtimes, or publishing
automation.

The plugin should at minimum:

- read ordered top-level `rules/*.md` relative to the plugin/package location
- expose an `experimental.chat.system.transform` hook that prepends or appends a
  clear Loom system instruction block from those rules
- expose small helper metadata or functions that make bundled `skills/` and
  optional `commands/` discoverability/fallback conclusions inspectable without
  pretending OpenCode has first-class registration APIs if it does not
- include a local structural smoke path that validates bundled rule reads without
  requiring a real OpenCode TUI session

Document in the fixture or `INSTALL.md`:

- npm plugin entry for normal users, as a package placeholder if package naming is
  not finalized
- local file/path plugin entry for cloned-repo users
- why Git URL plugin specs are not recommended
- whether rules, skills, and commands are plugin-first, fallback, or unresolved
  after this validation pass
