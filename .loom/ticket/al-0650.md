# Compound Simplification v2 Plan (No Backward Compatibility)

## Ticket Intent

Refactor `src/agent_loom/compound` to a minimal, orthogonal architecture that delivers value with less autonomous behavior and less code surface:

1. **Keep automated observations -> instincts** (this remains the only automatic learning lane).
2. **Make skills explicit-only** (no automatic skill writes; use deliberate skill-authoring flow).
3. **Decouple LOOM.md/derived docs maintenance** from instincts and skills.
4. **Remove dead code aggressively** (new project, no compatibility burden).

---

## Hard Decisions (Locked)

1. **No backward compatibility**
 - Remove obsolete commands/flags/paths immediately.
 - Delete proposal-era and direct-headless-skill-era code paths instead of deprecating.

2. **Orthogonality is non-negotiable**
 - Instincts pipeline is independent from skill creation.
 - Skill creation is independent from LOOM.md refresh.
 - LOOM.md refresh is independent from instincts extraction.

3. **Automatic mutation scope is narrowed**
 - Automatic path may write only:
 - `.loom/compound/runtime/observations.jsonl`
 - `.loom/compound/state.json`
 - `.loom/compound/instincts.json`
 - `.loom/compound/INSTINCTS.md` (if retained as derived index)
 - Automatic path must not write skills or broad docs.

4. **Skill compounding is explicit user/agent intent**
 - The only sanctioned skill creation/update path is explicit invocation (command/procedure) that uses skill-authoring guidance.
 - Harness nudges are allowed and encouraged; autonomous skill generation is removed.

5. **Dead code elimination is a delivery criterion**
 - If a module/symbol/flag/path has no active responsibility in the new architecture, it is removed in the same change set.

---

## Problem Statement (Current)

Current Compound combines too many concerns:

- observation ingestion,
- autonomous headless learning,
- proposal application,
- skills mutation,
- docs refresh/mirror behavior,
- evidence packaging (episodes/decisions/blobs),
- harness nudge policy.

This coupling creates:

- unclear mental model,
- excessive autonomous side effects,
- difficult correctness reasoning,
- larger dead-surface risk,
- adapter thickness drift.

---

## Target Architecture (Post-Simplification)

### Subsystem A — Observation Ingestion (automatic)

**Responsibility:** append normalized, redacted records to centralized observations log.

- Canonical file: `.loom/compound/runtime/observations.jsonl`
- Adapters:
 - OpenCode plugin (`.opencode/plugins/compound_engineering.ts`) -> capture + append + optional nudge
 - Claude hooks (`.claude/settings.json` -> `loom compound claude-hook`) -> capture + append + optional nudge

No skill writes. No LOOM.md writes.

### Subsystem B — Instinct Synthesis (automatic or scheduled)

**Responsibility:** derive/update instincts from new observation slices (and optional bounded git evidence), with cursor state.

- Inputs:
 - observations since cursor (`observations_offset_bytes` + `observations_prefix_sha256`)
 - optional cheap git signal (changed files + diffstat only)
- Outputs:
 - `.loom/compound/instincts.json`
 - optional `.loom/compound/INSTINCTS.md` index
 - `.loom/compound/state.json` cursor advancement

No skill writes. No LOOM.md writes.

### Subsystem C — Explicit Skill Compounding (manual)

**Responsibility:** create/update skills only when explicitly requested.

- Trigger: command/procedure intent (e.g. “create skill from this procedure”).
- Must read skill-authoring guidance before writing.
- Writes only skill files/mirrors intentionally.

No automatic trigger from session idle/stop.

### Subsystem D — Context/Docs Refresh (manual or explicit maintenance)

**Responsibility:** regenerate managed docs/blocks.

- `loom compound refresh` remains explicit maintenance step.
- Not part of instincts run.
- Not part of skill creation run.

## CLI Contract Redesign

### Keep

- `loom compound init`
- `loom compound refresh`
- `loom compound sync`
- `loom compound prime`
- `loom compound claude-hook` (ingestion + nudge only)

### Replace / Remove

- Remove `loom compound learn` current semantics as autonomous direct-headless skill generator.
- Remove `--proposals`, `--harness`, `--model` flags from learning path if command remains under same name.

### Introduce (recommended)

1. `loom compound instincts update`
 - Runs instincts synthesis from new observations.
 - Optional `--auto` for quiet adapter-triggered invocation.
 - No skill/doc mutation.

2. `loom compound skills author` (or equivalent explicit entrypoint)
 - Explicit skill creation/update flow only.
 - Must consume skill-authoring guidance.
 - May accept `--from-procedure` / stdin narrative.

3. `loom compound nudge` (optional helper)
 - Returns concise suggestion text when evidence suggests skill-worthy procedure.
 - Used by adapters to prompt user/agent, not mutate files.

No compatibility aliases required.

- `id`, `ts`, `harness`, `event`, `session_id`, `cwd`

Optional:

- Keep deterministic redaction and truncation.

- `.loom/compound/instincts.json`

Add/retain metadata fields enabling LLM awareness and drift handling:
- `source_observation_count`
- `last_observation_ts`
- `confidence`
- `domains/tags`
- `evidence_examples` (bounded)

### State Cursor (retain)

Primary file:
- `.loom/compound/state.json`

Keep only fields needed for incremental ingestion:
- `observations_offset_bytes`
- `observations_prefix_sha256`
- `observations_count`
- `observations_tail_sha256`
- `updated_at`

Remove fields tied to deleted episode/proposal workflow.

3. No automatic LOOM.md mutation from instincts updates.
4. No automatic skill mutation from instincts updates.

This provides awareness while preserving subsystem boundaries.
