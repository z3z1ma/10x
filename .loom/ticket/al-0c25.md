# Team YAML schema: communication policy + escalation contract

Objective alignment:
The sprint objective calls for explicit communication and escalation boundaries in YAML team composition. This ticket defines and codifies that contract so runtime behavior is deterministic and reviewable.

Scope:
- Extend `src/agent_loom/team/composition.py` schema parsing to support explicit escalation policy in `communication` (target + timeout/behavior).
- Validate tokens/roles/groups with deterministic normalization and actionable errors.
- Persist normalized escalation policy into run state shape used by runtime policy checks.
- Update/add fixture YAMLs in `tests/fixtures/team_composition/` for valid/invalid escalation configurations.
- Update parser/unit tests in `tests/test_team_composition_schema.py` and any direct policy tests that consume parsed roster spec.

Non-goals:
- No tmux behavior changes.
- No new CLI commands.
- No migration shim for older schema versions; choose one authoritative schema/version and enforce it.

Implementation plan:
1) Design final escalation schema (fields, required vs optional, defaults, validation constraints) and codify in parser constants/types.
2) Implement parser + serializer updates (`as_dict`) to preserve deterministic ordering/output.
3) Add strict validation for invalid roles/targets/timeouts with precise error paths.
4) Wire parsed escalation data into run JSON roster spec persistence path (start/resume invariants).
5) Add fixture + test coverage for happy path and failure modes.

Verification:
- uv run pytest tests/test_team_composition_schema.py tests/test_team_comms_policy.py tests/test_team_start_yaml_composition.py
- uv run ruff check .
- uv run basedpyright

Risks/edge cases:
- Ambiguity between built-in routes and custom role routes; must reject conflicting definitions.
- Escalation target cycles/invalid target tokens can silently weaken policy if not rejected.
- Schema versioning choice can break existing fixture assumptions; tests must pin expected version behavior.

## Acceptance Criteria

- Parser accepts valid escalation policy in YAML and emits normalized canonical structure in run roster spec.
- Invalid escalation definitions fail fast with path-specific error messages and hints.
- Composition + comms policy tests pass with new schema coverage and no flaky ordering assertions.
