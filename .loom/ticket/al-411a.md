## Root Cause Analysis

**Symptom**: Worker w2 sidecar crashes with "Model zai-coding-plan/glm-5 not found" despite run.json having correct model config.

**Evidence chain**:
1. run.json (lines 62-75) shows omp.models.worker = "github-copilot/claude-sonnet-4.5" ✓
2. sidecar.started event shows model = "zai-coding-plan/glm-5" ✗ (WRONG)
3. OMP exits with rc=1, capture shows "Model 'zai-coding-plan/glm-5' not found"

**Code path verified**:
- _model_for_role(run, role="worker", harness="omp") (core.py:3814-3827)
  - Precedence: run[harness].models[role] → run[harness].model → run.opencode
  - For worker+omp: should return run.omp.models.worker
- _respawn_active_worker_if_missing (core.py:3504-3610)
  - Line 3548: harness from run.get("harness")
  - Line 3583: model = _model_for_role(run, role, harness=harness)
- tui(...) (core.py:1353-2100)
  - Line 1389: model parameter passed through
  - Line 1958-1965: _omp_tui_argv(model=model, ...)
- _omp_tui_argv (core.py:1110-1130)
  - Line 1121-1122: argv += ["--model", model]

**Root cause hypothesis**:
The wrong model appears in the sidecar.started event, which means _model_for_role resolved the wrong value at spawn time. Possible causes:
1. run.json was stale/corrupted at respawn time
2. _model_for_role had a different harness parameter than expected
3. There was a race condition where run.json was updated between reads

**Impact**: This is NOT a logic bug in _model_for_role. The tests prove the function works correctly. The bug is likely:
- Timing issue during run.json updates
- Or incorrect harness parameter being passed during worker revival

**Deterministic repro**: Cannot reproduce with current run.json state because it has been fixed by the manager.
