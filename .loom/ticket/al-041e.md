# Team core: extract communication routing + send policy domain

Objective alignment:
Message routing, target policy checks, and delivery suggestions are high-change logic currently concentrated near the end of `src/agent_loom/team/core.py`. Extracting them into a domain module reduces hotspot churn and makes communication rules auditable.

Scope:
- Introduce a communication domain module (for example `src/agent_loom/team/communication_policy.py` or equivalent) for route policy, sender resolution, target authorization, and delivery suggestion helpers.
- Move communication-specific helper functions out of `team/core.py` while keeping `send`, `inbox_send`, and related public entrypoints behavior-stable.
- Keep target expansion behavior aligned with `src/agent_loom/team/targets.py` and existing communication-policy semantics.
- Update coverage in `tests/test_team_comms_policy.py`, `tests/test_team_send_broadcast.py`, and any affected `tests/test_team_cli_ux.py` cases.

Non-goals:
- No changes to durable inbox storage backend APIs in `team/inbox.py`.
- No new messaging features or route types.
- No tmux command wrapper redesign.

Implementation plan:
1) Extract communication helper cluster from core into a new module with explicit typed inputs/outputs.
2) Keep `team/core.py::send` as orchestration shell; delegate policy checks and route decisions to new module.
3) Ensure error codes/messages for denied routes and invalid senders remain stable.
4) Extend tests for policy denial, escalation path, broadcast routing, and offline delivery suggestion behavior.
5) Remove duplicated/unused communication helpers and imports from core.

Verification:
- uv run pytest tests/test_team_comms_policy.py tests/test_team_send_broadcast.py tests/test_team_cli_ux.py
- uv run ruff check .
- uv run basedpyright

Risks/edge cases:
- Behavior drift in route authorization can silently weaken communication constraints.
- Sender inference from env (`ROLE_WORKER`, worker_id) is brittle if moved without tests.
- Split across `targets.py` and new policy module can create duplicated normalization logic; enforce single normalization path.

## Acceptance Criteria

- Communication routing/policy helpers live in a dedicated domain module with `team/core.py` delegation.
- Send/inbox command behavior and error semantics remain stable under existing tests.
- Communication policy and broadcast tests cover extracted paths and pass.
