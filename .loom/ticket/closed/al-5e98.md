# Implement YAML-driven member lifecycle + BYO agent wrapping

Objective alignment:
The YAML schema becomes useful only when spawn/runtime behavior consumes it. This ticket makes team composition operational by mapping work to member profiles and allowing user agents without losing Loom protocol guarantees.

## Scope
- Use persisted composition to drive member role behavior at spawn time.
- Implement always-on vs ephemeral member handling.
- Implement worktree-pattern mapping to member configuration.
- Support BYO agent definitions while wrapping mandatory Loom instruction block.

## Non-goals
- No broadcast fan-out command changes (handled separately).
- No ticket sprint command changes (handled in ticket module ticket).

## Implementation plan
1. Add composition resolution helpers (likely in new module, not directly in `core.py`) for:
   - selecting member profile for a ticket/worktree
   - resolving always-on vs ephemeral constraints
   - deriving role/harness/model/agent defaults
2. Wire resolution into spawn lifecycle:
   - `src/agent_loom/team/core.py` (`spawn`, `resume_worker`, and integrator path if applicable)
3. Implement BYO agent wrapping policy:
   - compose user agent prompt with non-negotiable Loom protocol preamble
   - ensure wrapper is applied consistently across harnesses (`opencode`, `claude`, `omp`, `codex`)
4. Add tests:
   - `tests/test_team_spawn_yaml_members.py`
   - `tests/test_team_byo_agent_wrapping.py`
   - include conflict and missing-agent failure paths

## Verification
- `uv run pytest tests/test_team_spawn_yaml_members.py tests/test_team_byo_agent_wrapping.py`
- `uv run ruff check .`
- `uv run basedpyright`

## Acceptance criteria
- Spawn selects member configuration from YAML deterministically.
- Always-on and ephemeral member semantics are enforced by runtime logic.
- BYO agent prompts are wrapped with Loom protocol constraints before launch.
- Tests cover success paths and key failure cases.

## Risks and edge cases
- Risk: multiple members match the same worktree or ticket.
  - Detection: non-deterministic spawn config in tests.
  - Mitigation: deterministic tie-breaker or explicit error.
- Risk: wrapper prompts duplicate instructions or break harness-specific prompt expectations.
  - Detection: harness launch tests fail or generated prompt snapshots drift.
  - Mitigation: single wrapper composer with harness-specific adapters.
