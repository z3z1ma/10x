# Add first-class ticket sprint context commands

Objective alignment:
Current sprint behavior in ticket creation depends mainly on `TEAM_SPRINT_TAG` env injection. Ticket should own an explicit sprint context so workflows are deterministic outside Team and easier to reason about.

## Scope
- Add ticket CLI sprint context commands.
- Persist sprint context in ticket subsystem state.
- Use explicit sprint context for create/list defaults with clear precedence.

## Non-goals
- No changes to Team sprint lifecycle APIs beyond consuming ticket behavior as-is.
- No dashboard/UI integration in this ticket.

## Implementation plan
1. Add command surface in ticket CLI:
   - `loom ticket sprint show`
   - `loom ticket sprint set --name ... --tag ...`
   - `loom ticket sprint clear`
2. Implement storage and retrieval for sprint context in ticket module.
3. Update create/list flows to consume explicit sprint context first, then env fallback, then none.
4. Add and update tests:
   - extend `tests/test_ticket_ux.py`
   - add dedicated sprint-context tests in `tests/test_ticket_sprint_context.py`
5. Update ticket documentation for sprint-context workflow.

## Verification
- `uv run pytest tests/test_ticket_ux.py tests/test_ticket_sprint_context.py`
- `uv run ruff check .`
- `uv run basedpyright`

## Acceptance criteria
- Ticket CLI exposes sprint show/set/clear commands.
- Sprint context persists and is used by create/list defaults deterministically.
- Precedence between explicit sprint context and env tag is covered by tests.
- Documentation includes sprint-context usage examples.

## Risks and edge cases
- Risk: stale sprint context leaks into unrelated work.
  - Detection: create/list unexpectedly tagged outside sprint context tests.
  - Mitigation: explicit `clear` command and tests around default-empty state.
- Risk: precedence regressions break existing TEAM_SPRINT_TAG behavior.
  - Detection: existing create tests fail or tags differ unexpectedly.
  - Mitigation: add compatibility test coverage for env-only mode.
