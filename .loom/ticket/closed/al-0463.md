# Sprint prep: Public Launch Architecture Cleanup

Objective:
Improve code architecture, remove duplication. Ensure codebase is ready to go public.

## Sprint Brief

### Objective restatement
Stabilize Loom’s public command surface by removing repeated CLI plumbing and shrinking oversized modules that currently hide multiple responsibilities in single files.

### Sprint focus (2-5 words)
CLI Boundary Consolidation

### Proposed sprint name (2-5 words)
Launch-Ready CLI Core

### Why this sprint focus is the best next step
The largest architecture debt is concentrated in command entrypoints and orchestration modules that users hit first (`loom team`, `loom workspace`, `loom ticket`). This is where duplication and complexity directly impact maintainability, correctness, and public-facing UX consistency. Tightening these boundaries now gives the highest leverage before broader launch polish.

### Current state
- **Existing tickets that matter**
  - `al-0463` (this sprint prep) is the only ticket currently tagged for this sprint before planning.
  - `al-c09c` (open): pack diff flag work, adjacent but not core architecture cleanup.
  - Review queue contains architecture-heavy workspace tickets (`al-97e8`, `al-33c8`, `al-e024`, `al-1f81`, `al-0e62`) that may overlap with planned work and must be reconciled before implementation to avoid duplicate refactors.
  - `al-0650` (in_progress) is a large compound architecture effort; it should remain isolated from this sprint’s CLI-core refactor lane.
- **Codebase state that matters**
  - `git status --short --branch`: branch clean except untracked `.venv`.
  - Recent history focuses on team/compound capabilities (latest commit `869fef7 chore: compound`), with no dedicated launch-focused CLI architecture pass yet.
  - Hotspot modules by size/complexity:
    - `src/agent_loom/team/core.py` (~6404 LOC)
    - `src/agent_loom/workspace/cli.py` (~2534 LOC)
    - `src/agent_loom/team/cli.py` (~1900 LOC)
  - Duplicated CLI serialization/output helpers exist across modules (`team/cli.py`, `workspace/cli.py`, `ticket/cli.py`, `compound/cli.py`, `pack/cli.py`, `init.py`).

### Plan overview
1. Define launch CLI architecture boundaries and overlap guardrails with existing review-state tickets.
2. Extract shared CLI result/output primitives and migrate all CLI entrypoints to one contract.
3. Split oversized `team/core.py` into domain modules while preserving public behavior.
4. Break `workspace/cli.py` and `team/cli.py` into command-group modules with thin command dispatch entrypoints.
5. Add architecture guard tests/docs so structure does not regress.

### Risks + unknowns (and how we’ll resolve them)
- **Overlap risk with review-state tickets**: may duplicate already-completed refactors in other branches.
  - **Resolution**: first ticket performs overlap audit against review ticket scopes/files and defines authoritative ownership.
- **Behavior drift risk during module splits**: CLI output/exit-code regressions are easy to introduce.
  - **Resolution**: preserve command contract via existing test suites plus targeted CLI UX regression tests per refactor ticket.
- **Refactor blast radius**: large-file decomposition can stall if attempted as one change.
  - **Resolution**: sequence into scoped, dependency-driven tickets with clear non-goals and explicit verification gates.
