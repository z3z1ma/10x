# Sprint prep: Modular refinement

## Sprint Brief

### Objective restatement
Deliver a concrete modularity pass on the Team subsystem by reducing `src/agent_loom/team/core.py` hotspot complexity while preserving CLI behavior and operational semantics.

### Sprint focus (2-5 words)
Team Core Decomposition

### Why this sprint focus is the best next step
`team/core.py` is currently ~7.2k LOC and remains the dominant coupling hotspot in the repository. Ongoing feature work (YAML roster/runtime, sprint orchestration, messaging) keeps landing in this file, increasing merge risk and slowing review. The highest-leverage next step is domain extraction by responsibility (objective/sprint, communication policy, worker planning, wait/check-in loop), with parity tests to prove no behavior regressions.

### Current state
- Existing tickets that matter:
  - Open backlog already contains a prior sprint set (`al-445a` and children `al-0c25`, `al-2510`, `al-30d0`, `al-b1cb`, `al-92ed`) focused on YAML/sprint foundations.
  - `al-c09c` remains open and unrelated to Team-core modularity.
  - This sprint prep ticket: `al-1696` (tag `sprint:Modular-refinement`).
- Codebase state that matters:
  - `git status --short --branch`: main is clean except new ticket artifact (`?? .loom/ticket/al-1696.md`) plus tickets created during prep.
  - Recent direction (`git log -n 20 --oneline`) shows continued Team/YAML orchestration work (`feat(team): load and persist YAML composition at start`, `enforce broadcast targets and comms policy routes`, `apply YAML member resolution...`).
  - Module hotspot metrics:
    - `src/agent_loom/team/core.py`: 7264 LOC
    - `src/agent_loom/team/cli.py`: 1092 LOC
    - Next largest team modules are materially smaller (`prompts.py` 804, `composition.py` 636).
  - `team/AGENTS.md` explicitly calls out `team/core.py` as a decomposition hotspot and sets guardrails for command-handler thinness and import direction.

### Plan overview
1) Extract objective/sprint state orchestration from core into a dedicated domain module with stable delegations.
2) Extract communication routing/policy helpers from core into dedicated domain module while preserving send/inbox behavior.
3) Extract worker/profile planning primitives used by spawn-family flows into dedicated domain module.
4) Isolate wait/check-in + capture loop operational helpers into dedicated module.
5) Run a fan-in regression pass to enforce parity and architectural boundaries.

### Risks + unknowns (and how we'll resolve them)
- **Risk: behavior drift during extraction** (especially sprint-tag precedence, sender/route authorization, spawn headcount).
  - Resolution: ticket-level regression suites pin command-observable behavior before/after each extraction.
- **Risk: merge conflicts from shared hotspot** (`team/core.py`) across multiple workers.
  - Resolution: enforce dependency ordering so refactors land incrementally with rebases between waves.
- **Risk: circular imports when introducing new domain modules.**
  - Resolution: keep module contracts narrow (pure helpers + typed inputs) and retain side-effect orchestration in core.
- **Unknown: hidden coupling to pre-existing helper side effects.**
  - Resolution: final fan-in ticket (`al-2bfa`) runs broad Team regression matrix and captures residual hotspots as follow-up tickets.
