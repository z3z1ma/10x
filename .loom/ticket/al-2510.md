# Team runtime: member lifecycle/workspace mapping from YAML

Objective alignment:
The objective explicitly calls for mapping team members to worktree patterns and distinguishing always-on vs ephemeral members. This ticket implements runtime resolution rules so member lifecycle/workspace behavior is intentional and enforceable.

Scope:
- Extend member schema/runtime mapping to support explicit workspace/worktree mapping data for custom members where needed.
- Update `src/agent_loom/team/composition_runtime.py` and spawn flow in `src/agent_loom/team/core.py` to consume mapping deterministically.
- Ensure lifecycle enforcement remains strict (worker ephemeral, manager/architect/integrator always-on unless intentionally changed by schema contract).
- Add/adjust tests in `tests/test_team_spawn_yaml_members.py`, `tests/test_team_targets.py`, and any spawn/integrator tests impacted by workspace mapping.

Non-goals:
- No new role types beyond what schema allows.
- No changes to unrelated workspace subsystem APIs.
- No broad refactor of team/core decomposition outside required touch points.

Implementation plan:
1) Define runtime mapping contract for member->workspace/worktree-key derivation (explicit field vs derived fallback).
2) Implement mapping resolution in composition_runtime with clear precedence and validation errors.
3) Wire spawn paths to honor resolved workspace/worktree values and preserve mount safety checks.
4) Update target resolution/broadcast behavior if member addressing relies on new mapping metadata.
5) Add regression tests for always-on and ephemeral members across repo_root/worktree workspaces.

Verification:
- uv run pytest tests/test_team_spawn_yaml_members.py tests/test_team_targets.py tests/test_team_spawn_integrator.py tests/test_team_spawn_mounts.py
- uv run ruff check .
- uv run basedpyright

Risks/edge cases:
- Two members resolving to same worktree key can cause collisions; must detect and reject.
- Invalid workspace mapping can create path traversal/security problems; keep strict normalization and repo-root constraints.
- Lifecycle enforcement and custom mapping may conflict; errors must explain remediation clearly.

## Acceptance Criteria

- Spawn/runtime uses YAML-defined lifecycle + workspace mapping consistently for built-ins and custom members.
- Invalid/ambiguous mapping is rejected before spawn side effects occur.
- Updated tests prove deterministic resolution and no regression in existing spawn/mount semantics.
