# Team CLI: roster inspect/validate command set

Objective alignment:
YAML-defined teams need first-class operator ergonomics. Fast validation/introspection before `team start` reduces misconfiguration and speeds iteration.

Scope:
- Add CLI entrypoints under `loom team` for roster inspection/validation (for example validate and show-normalized output).
- Implement command handlers in `src/agent_loom/team/commands/` and wire parser in `src/agent_loom/team/cli.py`.
- Reuse `composition.py` parser to avoid duplicate validation logic.
- Add tests for command UX + JSON output in `tests/test_team_cli_ux.py` (or new dedicated roster CLI test module).

Non-goals:
- No changes to spawn orchestration semantics.
- No dashboard/server UI work.
- No support for non-YAML roster formats.

Implementation plan:
1) Define subcommand names, args, and json/text output contract.
2) Implement command plumbing and error mapping to existing TeamError patterns.
3) Expose normalized roster projection for inspection mode.
4) Add tests for success and failure cases (missing file, invalid yaml, invalid schema, valid normalized output).

Verification:
- uv run pytest tests/test_team_cli_ux.py tests/test_team_composition_schema.py tests/test_team_start_yaml_composition.py
- uv run ruff check .
- uv run basedpyright

Risks/edge cases:
- Output churn can destabilize downstream scripts; keep JSON contract explicit in tests.
- Duplicate parsing paths can diverge; command must delegate to single parser surface.

## Acceptance Criteria

- `loom team` exposes roster validation/introspection commands with documented argument behavior.
- Valid roster exits 0 and can emit normalized JSON; invalid roster exits non-zero with actionable errors.
- CLI test coverage locks output contract for both human-readable and JSON modes.
