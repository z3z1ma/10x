# Add team start-time model validation and recovery workflow for OMP

Prevent invalid model config from being persisted into run.json and document deterministic recovery commands for existing runs.

## Acceptance Criteria

Team start rejects known-invalid OMP model selections with actionable guidance, and docs/tests cover recovery for existing runs.

## Notes

**2026-02-15T19:36:17Z**

Execution brief:

Objective alignment
- al-44f5 prevents runtime crash loops, but we also need to prevent creation of bad run configs in the first place so future workers do not regress into the same outage pattern.

Scope
- Add OMP model validation at config-write time in `loom team start` when explicit model overrides are provided.
- Surface actionable CLI errors before persisting invalid model values.
- Document and test the recovery workflow for already-persisted invalid run configs.

Non-goals
- No validation enforcement for non-OMP harnesses in this ticket.
- No automatic mutation of existing run.json without explicit user command.
- No tmux/session orchestration redesign.

Implementation plan
1) In `src/agent_loom/team/core.py::start(...)`, validate user-supplied OMP model overrides before `save_run(...)` when any of these flags are provided:
   - `--model`, `--manager-model`, `--worker-model`, `--investigator-model`, `--integrator-model`.
2) Reuse shared helper introduced by al-44f5 (or extract if needed):
   - known-invalid model -> raise `TeamError` with remediation command example using `omp --list-models <pattern>` and `loom team start <team> --worker-model <valid>`.
   - indeterminate provider/listing failure -> warn and proceed (do not hard-fail).
3) Add tests:
   - `tests/test_team_harness_omp.py`: start rejects definitely-invalid override.
   - `tests/test_team_cli_ux.py` or related CLI tests: error text includes remediation guidance.
4) Update docs:
   - `src/agent_loom/team/README.md` add OMP model troubleshooting and recovery steps for existing runs.

Verification commands
- uv run ruff check .
- uv run basedpyright
- uv run pytest tests/test_team_harness_omp.py
- uv run pytest tests/test_team_cli_ux.py

Acceptance criteria
- `loom team start ... --harness omp --worker-model <invalid>` fails fast with actionable guidance.
- Invalid OMP model overrides are not persisted to run.json.
- Documentation includes deterministic recovery commands for already-broken runs.
- New tests cover reject path and message quality.

Risks / edge cases
- False negatives from model listing API can block valid private models; distinguish definitive "no match" from listing failure and avoid hard-failing ambiguous cases.
- Existing runs may still contain invalid models; doc recovery steps must be explicit and tested.
