# Workspace: unify git plumbing layer (dedupe git_ops/diff_ops/core.git)

Workspace implements a substantial amount of git plumbing (`git_ops.py`, plus overlapping helpers in `diff_ops.py` and `repo_ops.py`). Meanwhile `agent_loom.core.git` contains a second, partially-overlapping git helper layer.

Result: duplicated logic, inconsistent error handling, and unclear layering.

## Design

Design goal: a single git plumbing layer with a clear API boundary.

Plan:
- Define the canonical low-level git helpers in `src/agent_loom/core/git.py` (or a new `src/agent_loom/core/git_ops.py` if we want to preserve the existing core API).
- Workspace should depend on that layer for:
  - locating git dirs / repo roots robustly (worktree-aware)
  - parsing `git worktree list --porcelain`
  - ref existence checks and merge-base
  - fetching/cloning (with shallow support)
  - consistent `run()` behavior and error reporting
- Consolidate duplicate helpers:
  - `workspace.git_ops.git_ref_exists` vs `workspace.diff_ops.git_ref_exists`
  - merge-base/ref parsing helpers
  - `repo_ops.repo_fetch_ref` vs `git_ops.git_fetch_ref`

Implementation constraints:
- Keep workspace semantics stable (guardrails, safety checks, shallow clone behavior).
- Keep output deterministic and preserve tests.

Non-goals:
- No new features.
- No CLI changes.

## Acceptance Criteria

- There is exactly one implementation of each git helper concept (ref-exists, merge-base, worktree porcelain parsing, repo root resolution).
- `src/agent_loom/workspace/git_ops.py` is removed or becomes a thin adapter around `agent_loom.core.git`.
- Gates pass: `uv run basedpyright`, `uv run ruff check .`, `uv run pytest tests/test_workspace_*.py`.
