# Workspace: dial in polyrepo harness value prop + naming (poly vs harness)

`workspace poly` is already a polyrepo control plane (inventory, group worktrees, exec, snapshots, services/deps), which is essentially a *harness*.

The current naming (`poly`) under-sells what it is and doesn’t clearly differentiate the value over 'just git'.

## Design

Work items:
- Pick terminology:
  - Internal: 'harness' (control plane), 'workspace' (the whole subsystem), 'repo mode' (single-repo).
  - CLI: keep `poly` for backwards compatibility in tests, but add an alias `harness` (preferred) OR update docs to call `poly` the harness.
- Update the cookbook (`src/agent_loom/workspace/README.md`) to lead with:
  - What you get beyond git: annotations, TTL, sandboxes, safe cleanup/GC, deterministic selection/intent, cross-repo exec + context, services/deps context.
  - User stories (3-5): sprint group worktrees, incident sandbox, agent fanout, cleanup day.
- Update `loom workspace --help` descriptions to mention harness.

Non-goals:
- No new runtime features beyond alias/help/doc.

## Acceptance Criteria

- Cookbook and help text clearly articulate value proposition and mental model.
- If an alias is added, tests cover it (CLI output contract).
