# Workspace: optional canonical locations for harness roots/clones/worktrees

Canonical locations are low-to-medium value, but they can reduce friction for repeated usage:
- users want a predictable place where harnesses live
- CI/agents benefit from stable paths

Today poly mode is anchored to the current directory, and repo mode anchors to the repo.

## Design

Options to explore:
- `loom workspace harness init --root <path>` (explicit location)
- A simple registry of known harness roots under `~/.loom/harnesses.json` (or similar), allowing:
  - `loom workspace harness use <name>`
  - `loom workspace harness ls`
- Configurable defaults in `workspace.json` (e.g., repo/worktree dirs under the root are already configurable).

Guardrails:
- Never create or delete outside the chosen root without explicit flags.

Non-goals:
- No automatic discovery across the filesystem.

## Acceptance Criteria

- A decision is recorded (ADR or doc note) on whether canonical locations are worth it.
- If implemented, includes tests and no nondeterministic absolute-path output in contract surfaces.
